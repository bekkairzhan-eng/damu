# Техническое задание — Фронтенд BI Damu

**Версия:** 1.0
**Проект:** BI Damu — платформа карьерного развития ИТР-сотрудников BI Group
**Аудитория:** Frontend-разработчики
**Текущее состояние:** React-прототип готов, все основные экраны реализованы на хардкоде/localStorage. Задача — подключить реальные API (см. [BACKEND_SPEC.md](BACKEND_SPEC.md)) и дореализовать экраны, которых пока нет в прототипе (раздел 9 этого документа).

## 1. Стек и структура проекта

| Компонент | Технология |
|-----------|-----------|
| Фреймворк | React 19 + Vite |
| Роутинг | React Router v7 |
| Язык | JavaScript (TypeScript не используется) |
| Стили | Инлайн `style={{}}`. CSS-классы — только для иконок `material-symbols-outlined` и hover-классов сайдбара (через тег `<style>`). Tailwind подключён, но почти не используется |
| Состояние | `ProfileContext` (глобальное) + `useLocalStorage` (персистентность) + локальный `useState` |

### Точка входа и лэйаут

- `src/App.jsx` — `ProfileProvider` + маршруты React Router
- `src/components/Layout.jsx` — сворачиваемый сайдбар + `<Outlet />`; `TopNav.jsx` вынесен отдельно
- `src/ProfileContext.jsx` — `overallScore` (рейтинг профиля), `isDark`/`toggleDark`
- `src/hooks/useLocalStorage.js` — используется вместо `useState` там, где нужна персистентность между перезагрузками

### Стилевые соглашения

- Палитра: основной `#4361ee`, тёмный `#0f1923`, вторичный текст `#7a8fa0`, граница `#e8edf2`, поверхность `#fff`, фон `#f5f6fa`

## 2. Роли и доступ

| Роль | Что видит |
|------|-----------|
| `employee` | Профиль, планы, навыки, карьерный трек. Запрашивает аттестацию, подтверждает навыки, подаёт заявки на обучение |
| `hr` | Панель `/hr` — очередь аттестаций, ввод оценки, (новое) добавление сертификатов сотрудникам |
| `admin` | Панель `/admin` — справочники: навыки, должности, карьерный граф, требования, веса рейтинга, пользователи/роли |

- Роли — из Keycloak-токена в проде; в прототипе — `localStorage auth:roles`, хук `useRole`
- `RoleGuard` на фронте — **только UX** (скрыть меню/роут), не защита. Сервер обязан проверять роль независимо от клиента ([BACKEND_SPEC.md, раздел 3](BACKEND_SPEC.md))
- Пользователь без нужной роли, зашедший на `/hr` или `/admin`, видит экран «Нет доступа»
- Пользователь с несколькими ролями видит переключатель панелей в TopNav
- HR в сайдбаре HR-панели видит ссылки «Мой профиль» и «Панель администратора»; Admin — «HR-панель» и «Мой профиль»

## 4. Ключевая логика существующих экранов

### CareerMap (`src/pages/CareerMap.jsx`)

Самый большой файл. Режим `buildTo === null` — стандартный вид (Foreman A → B → цель по центру); `buildTo !== null` — режим исследования, DFS по `CAREER_GRAPH`.

- `CURRENT_POSITION`, `CAREER_GRAPH`, `NEXT_GOAL_OPTIONS`, `GOAL_TO_PLAN_ID` — сейчас хардкод, уходят на `GET /api/career-graph?clusterId=`
- `findAllPaths` (DFS) + `computeLayout` (разбивка на prefix/branches/suffix) + `ForkConnector` (SVG-развилки) — чистая фронтенд-логика, бэкенд не участвует
- Кнопка «Установить карьерную цель» — только на карточках из `NEXT_GOAL_OPTIONS[CURRENT_POSITION]`, не текущая цель. Диалог подтверждения — локальный `confirmGoal`

### MyPlans / CareerPlanDetail

- `MyPlans/index.jsx` — читает `careermap:goal` из localStorage, динамически формирует название плана. Уходит на `GET /api/me/plans`
- `CareerPlanDetail.jsx` — три вкладки (требования, план обучения, готовность к оценке). Вкладка готовности считает обязательные пункты из `LEARNING_PLAN`, исключая `'Рекомендуемые курсы'` и пункты с `'казахский'` в названии

### Ключи localStorage (уйдут при переходе на API/сессию)

| Ключ | Где | Назначение |
|------|-----|-----------|
| `careermap:goal` | CareerMap, MyPlans, CareerPlanDetail | Текущая карьерная цель |
| `careermap:zoom` | CareerMap | Масштаб карты |
| `careermap:tour-seen` | CareerMap | Онбординг-тур завершён |
| `theme:dark` | ProfileContext | Тёмная тема |
| `auth:roles` | useRole | Роли пользователя (заменится на Keycloak-токен) |

## 6. Состояния экрана (сейчас отсутствуют в прототипе)

Каждый экран с данными из API должен реализовать три состояния:
- **Загрузка** — скелетон/спиннер, не пустой экран
- **Пусто** — например, нет навыков на подтверждение, нет уведомлений — не путать с ошибкой
- **Ошибка загрузки** — с возможностью повтора запроса

## 7. Требования ИБ (фронтенд-часть)

- Токен Keycloak — в памяти или httpOnly cookie, **не в localStorage** (XSS)
- `RoleGuard` — скрывает меню/роуты, но не заменяет серверную проверку. Не полагаться на него как на защиту данных
- Не показывать чужие данные по прямому URL-параметру (`employeeId` в query/body игнорируется бэкендом — приходит из токена), но и на фронте не строить ссылки на чужие профили без проверки роли
- Файлы (сертификаты, аватары) — только по подписанным S3-ссылкам с TTL, не хранить прямые постоянные URL

## 9. Новое: три сущности UnityBPM — что нужно фронтенду

Бизнес-флоу подробно разобран в [BPM_INTEGRATION.md](BPM_INTEGRATION.md). Ниже — что из этого требует изменений/новых экранов на фронте.

### 9.1 Подтверждение навыка — изменение существующего экрана

`MySkills` — при отображении статуса `rejected` у навыка теперь всегда приходит непустой `comment` (он обязателен на стороне BPM). Показать его сотруднику как причину отказа, а не просто статус «Отклонён» без пояснения.

### 9.2 Аттестация — новый статус и новое сообщение

`Assessment` (история) и виджет на `MyDashboard`:

- Новый промежуточный статус **«Ожидает аттестацию»** между «На рассмотрении у HR» и «Завершена» — участники комиссии добавлены в BPM, но встреча ещё не проведена. Дату/место система не знает (назначается HR офлайн через Outlook) — не пытаться показать дату, только сам статус
- При `outcome = passed` — помимо «Прошёл аттестацию», нужно показать пояснение **«На скамейке»** (ждёт открытую позицию) — текстовый бейдж/тултип, не блокирующий остальной интерфейс
- `AssessmentEntry` (HR, ввод оценки) — без изменений в форме, но после сохранения результата бэкенд теперь также решает `outcome`; фронту достаточно показать сообщение об успешном сохранении, никакой новой логики на этом экране не требуется

### 9.3 Заявка на обучение — два новых экрана

Экранов сейчас нет в прототипе.

**А. Сотрудник — подача заявки**
- Триггер: предупреждение об истечении/скором истечении сертификата на `ExperienceProfile` (в блоке сертификатов) — кнопка **«Записаться»**
- Также должна быть доступна самостоятельная подача (не только по истечению) — где именно разместить точку входа (`MySkills`? отдельный пункт меню?) — решить при дизайне, сейчас не определено
- Модалка: выбор программы обучения → `POST /api/me/training-requests`
- Новая вкладка/секция истории заявок (аналог `Assessment` для аттестаций) — статусы `pending` / `matched` / `completed` / `rejected` / `withdrawn`, из `rejected`/`withdrawn` — кнопка «Подать заново»
- Отзыв заявки («Отозвать») происходит в BPM, не на фронте Damu — на этом экране только отображение статуса, действия отзыва здесь не будет

**Б. HR-панель — добавить сертификат сотруднику**
- Новый экран/модалка в `/hr`: выбрать сотрудника, заполнить сертификат (название, кем выдан, дата получения, дата истечения, файл)
- Перед сохранением — вызвать `GET /api/hr/training-requests/match-check`; если есть совпадение — показать HR диалог **«У сотрудника уже есть поданная заявка. Подтвердить этим сертификатом?»** с выбором: привязать к существующей заявке / завести отдельно
- `POST /api/hr/certificates` с (или без) `linkToTrainingRequestId`

**Профиль опыта (`ExperienceProfile`)** — блок сертификатов должен отображать дату получения с разбивкой по кварталам (Q1–Q4), независимо от источника (LMS или ручной ввод HR).
