# BI DAMU — Интеграция с UnityBPM

Сводный документ по всем сущностям, которые Damu создаёт/получает через UnityBPM. Детали архитектуры (аутентификация, ручки, схемы БД) — в [BACKEND_SPEC.md](BACKEND_SPEC.md). Здесь фиксируется бизнес-логика и статусные модели по каждой сущности.

**Философия:** Damu — витрина и хранилище данных развития. Вся операционка (задачи, апрувы, эскалации, напоминания, переназначение при увольнении) — в UnityBPM. Утверждающим/координаторам не нужно заходить в Damu, чтобы обработать задачу — они работают внутри BPM. Damu узнаёт о результате через вебхук.

---

## Три сущности

| # | Сущность в BPM | Кто инициирует | Кто исполняет в BPM | Статус в BPM |
|---|----------------|-----------------|----------------------|--------------|
| 1 | Запрос на подтверждение навыка | Сотрудник (уровень 3–4) | Руководитель / эксперт / HR — по матрице | **Новый** — создаётся с нуля по этому ТЗ |
| 2 | Запрос на аттестацию | Сотрудник | HR-партнёр подразделения | **Новый** — создаётся с нуля по этому ТЗ |
| 3 | Заявка на обучение | Сотрудник или HR | Координатор Корпоративного университета | **Уже существует** — задача аналитика подключиться к текущему процессу, не проектировать заново |

## 1. Подтверждение навыка

> **Новый процесс.** В BPM его сейчас нет — создаётся заново по этому описанию.

Каждый навык на уровне 3 (Продвинутый) и 4 (Эксперт) требует апрува. Уровни 1–2 сотрудник подтверждает сам, без BPM.

### Флоу
1. Сотрудник нажимает «Подтвердить навык» в Damu
2. Damu передаёт `skill_id + employee_id` в BPM — BPM сам определяет утверждающего (эксперта) по матрице (орг. структура × навык)
3. BPM ставит статус **Ожидает подтверждения**, задача уходит эксперту
4. У эксперта в BPM две кнопки: **«Подтвердить»** / **«Отклонить»**
   - «Подтвердить» — без комментария, мгновенно
   - «Отклонить» — **комментарий обязателен** (валидация на стороне формы BPM, без комментария отправить нельзя)
5. BPM шлёт вебхук в Damu → статус меняется на **Подтверждён** / **Отклонён**; при отклонении комментарий эксперта приходит вместе с решением и отображается сотруднику как причина отказа

### Статусы (`employee_skill.status`)
`self` → `confirmed` (сразу, без апрува)
`pending` → `confirmed` / `rejected` (через BPM)

### Переназначение
При увольнении утверждающего BPM сам переназначает все активные задачи новому человеку. На стороне Damu изменений не требуется — результат придёт через тот же вебхук от нового утверждающего.

### Вебхук
`POST /webhooks/bpm/skill-approval` — см. [BACKEND_SPEC.md:1191](BACKEND_SPEC.md#L1191)

⚠️ Payload в BACKEND_SPEC.md сейчас показывает `comment` как опциональное пустое поле — нужно обновить контракт: `comment` обязателен, когда `decision = rejected`.

## 2. Запрос на аттестацию

> **Новый процесс.** В BPM его сейчас нет — создаётся заново по этому описанию.

### Условие доступности
Кнопка «Запросить аттестацию» активна только если % выполнения плана ≥ **85%** и ключевые навыки подтверждены.

### Инициатор
Сотрудник, подавший заявку через Damu. В документе, который падает HR, должно быть видно не только сам факт запроса, но и достаточно данных, чтобы HR понял степень готовности, не открывая ничего вручную:

| Поле | Источник |
|------|----------|
| ФИО, фото | HRMS (через Damu) |
| Текущая должность / роль | HRMS |
| Подразделение / кластер | HRMS |
| Целевая должность (куда хочет) | `career_plan.target_position_id` |
| % выполнения текущего плана | Damu (расчёт) |
| Статус ключевых навыков плана (навык → уровень → подтверждён/не подтверждён) | Damu (`employee_skill`) |
| Дата подачи заявки | Damu |
| Кол-во предыдущих попыток аттестации + комментарий HR по итогам последней (если не первая попытка) | Damu (`assessment`, `assessment_skill`) |

Всё это должно уйти в payload заявки при её создании, а не подтягиваться HR вручную из другой системы.

### Флоу в BPM
1. Заявка приходит HR-партнёру подразделения (кто именно — определяется по матрице ответственных, см. ниже) в раздел **«Мои документы»**, статус **«На исполнении»**
2. Изначально доступна только одна кнопка — **«Добавить участников»**. Кнопок «Исполнить» и «Отказать» на этом этапе нет вообще — заявку на этом шаге отклонить нельзя
3. По нажатию открывается модальное окно, HR выбирает участников комиссии
4. После того как участники добавлены, появляется кнопка **«Исполнить»** (кнопки «Отказать» по-прежнему нет — весь путь заявки: добавить участников → исполнить)
5. По нажатию «Исполнить» — это шаг **целиком внутри BPM, без участия Damu**:
   - каждому добавленному участнику создаётся задача **«Ознакомление»**
   - BPM сам рассылает участникам письмо своими средствами: «Вы были добавлены HR в список для проведения Ассесмента [ФИО сотрудника, должность, дата]»
   - интеграция с Outlook — не в этой итерации; BPM использует свой канал/шаблон рассылки, объединение с Outlook — отдельная задача на будущее
   - Damu в этом шаге не участвует и ничего не передаёт — письмо и задача участнику формируются и уходят полностью на стороне BPM
6. Саму встречу (аттестацию) HR назначает **офлайн, через Outlook** — приглашает и участников комиссии, и самого сотрудника. Damu дату/место не получает — вебхука с датой/местом нет. Пока встреча не проведена, сотрудник в Damu видит статус **«Ожидает аттестацию»**
7. На встрече (или сразу после неё) HR открывает уже существующий экран в Damu (`AssessmentEntry` — по каждому навыку прошёл/не прошёл + комментарий, кнопка «Начать/Завершить оценку») и вносит результат. Это действие целиком в Damu — новой интеграции с BPM для этого шага не требуется, экран уже есть
8. Как только HR завершил ввод, сотрудник получает уведомление в Damu:
   - если все навыки прошли — статус **«Прошёл аттестацию»** с пояснением **«на скамейке»** (ждёт открытую позицию — дальше HR/компания ищут место полностью офлайн, вне системы)
   - если есть непройденные навыки — существующий флоу: комментарии HR по навыкам, план остаётся `active`
9. Damu обновляет план развития — показывает только последнюю попытку; попыток не ограничено

### Статусы (`assessment.status`)
`pending` (заявка подана, участники не добавлены)
→ `awaiting_assessment` / «Ожидает аттестацию» (участники добавлены, встреча назначена HR офлайн)
→ `completed` (HR внёс результат в Damu)

Внутри `completed` — если все навыки в `assessment_skill` прошли, сотрудник дополнительно помечается как **«на скамейке»**; если есть непройденные — статус "на скамейке" не выставляется, план остаётся активным с комментариями.

### Открытый вопрос
- [ ] Снятие статуса «на скамейке» формально не определено. Вероятно, разрешается само собой, когда HRMS присылает новую `position_id` (сотрудник реально получил должность) — тогда план закрывается (`replaced`), создаётся новый под следующую цель. Нужно решение бизнеса: достаточно ли этого, или нужен отдельный ручной способ снять статус раньше (например, если поиск позиции затянулся).

### Матрица ответственных (HR по подразделению)
На стороне BPM есть универсальная матрица ответственных — она уже работает как «процессер» для других систем: при любом изменении (смена HR-партнёра на подразделении и т.п.) событие рассылается во все системы-потребители. Damu сейчас в их число не входит и должна стать ещё одним потребителем этого фида, чтобы:
- корректно определять, кому из HR показать заявку в «Мои документы» (или хотя бы отображать актуального HR-партнёра в Damu для сверки)
- не полагаться на устаревшие данные, если HR подразделения поменялся

Формат фида (push при каждом изменении / периодический снапшот / API по запросу) — вынести отдельным пунктом в контракт с BPM.

### Вебхук
`POST /webhooks/bpm/assessment-status` (опционально, для промежуточного статуса) — см. [BACKEND_SPEC.md:1220](BACKEND_SPEC.md#L1220)
Результат — не вебхук, а прямой ввод HR в Damu: `POST /api/hr/assessments/:id/result`

## 3. Заявка на обучение

> **Уже существует в BPM.** В отличие от двух предыдущих сущностей, это не новый процесс — сущность «Заявка на обучение» в BPM уже есть и используется. Ниже — требования и логика **со стороны Damu** (что нужно от процесса, чтобы Damu могла показать статус в профиле опыта). Задача аналитика — не проектировать процесс заново, а сверить с BPM-командой, что из описанного ниже уже поддерживается текущей сущностью, а что нужно доработать (в первую очередь: отдаёт ли процесс события по созданию/смене статуса заявки, и есть ли в нём поля `employee_guid` / `program_guid` для матчинга с LMS).

В отличие от двух предыдущих — заявка ≠ факт обучения. Между подачей заявки и реальным зачислением может пройти 1–2 месяца (набор группы), поэтому это не разовое решение "апрув/отказ", а сверка с LMS во времени.

### Триггеры подачи
- Истечение/приближение истечения сертификата — Damu показывает предупреждение, сотрудник жмёт «Записаться»
- Самостоятельная подача сотрудником (не обязательно по истечению)
- HR подаёт заявку за сотрудника — тот же путь, что и подача сотрудником, разница только в инициаторе

### Флоу
1. Сотрудник или HR подаёт заявку в Damu → создаётся запись `training_request` (`employee_guid`, `program_guid`, статус `pending`) → уходит в BPM
2. Раз в сутки крон сверяет каждую `pending`-заявку с LMS (BILIM) по паре `employee_guid + program_guid`
3. Если найден матч в LMS — статус меняется на `matched`, а после фактического завершения обучения в LMS — на `completed`, сертификат подтягивается в профиль опыта
4. Если BPM отклонил заявку (например, курс отменён / не набрали группу) — статус `rejected`
5. Если сотрудник сам отозвал заявку в BPM (кнопка «Отозвать») — статус `withdrawn`
6. Из `rejected` / `withdrawn` можно подать новую заявку — попыток не ограничено, каждая попытка — отдельная запись в истории

### Статусы (`training_request.status`)
`pending` → `matched` → `completed`
`pending` → `rejected`
`pending` → `withdrawn`

### Ручное добавление сертификата HR-ом
Отдельная функция в HR-панели — не через BPM, для случаев, когда обучение прошло полностью офлайн (мимо LMS).

- Перед созданием новой записи система проверяет: есть ли у сотрудника уже поданная `pending`-заявка по этому же курсу
- Если есть — HR получает предложение: «У сотрудника уже есть поданная заявка. Подтвердить этим сертификатом?» → выбор:
  - **Да** — привязать сертификат к существующей заявке, закрыть её как `completed`
  - **Нет** — завести сертификат как независимую запись (заявки не было)

### Назначение
Все эти данные (заявка + сертификат, откуда бы он ни пришёл — LMS или ручной ввод) отображаются в «Профиль опыта» с датой получения, с разбивкой по кварталам (Q1/Q2/Q3).

### Открытые вопросы (сверить с BPM-командой по факту существующего процесса)
- [ ] Какие поля и статусы уже есть в текущей сущности «Заявка на обучение» — совпадают ли со статусной моделью выше (`pending` / `matched` / `completed` / `rejected` / `withdrawn`), или нужно мапить на то, что уже реализовано?
- [ ] Отдаёт ли текущий процесс события наружу (вебхук/Kafka) при создании и смене статуса заявки, или Damu только читает состояние по запросу?
- [ ] Есть ли в сущности поля `employee_guid` / `program_guid` (или их аналоги), по которым можно сверяться с LMS — если нет, их нужно добавить
- [ ] Формат сверки крона с LMS: BPM/LMS отдаёт API для запроса по паре GUID, или это выгрузка списком?
- [ ] Что если заявка висит в `pending` без матча дольше разумного срока (например, 3+ месяца) — есть ли уже в процессе автоматический timeout, или остаётся вечно pending до ручного вмешательства?
- [ ] Поддерживает ли существующий процесс отзыв заявки сотрудником («Отозвать») — если да, как называется этот статус на его стороне?

## Общие механизмы

- **Аутентификация вебхуков:** `X-BPM-Secret` в заголовке, секрет в env — см. [BACKEND_SPEC.md:1187](BACKEND_SPEC.md#L1187)
- **Retry:** политика повтора при недоступности Damu на стороне BPM — уточнить у команды BPM (открытый пункт контракта, [BACKEND_SPEC.md:1504](BACKEND_SPEC.md#L1504))
- **Audit log:** каждое действие (заявка создана/отклонена/отозвана, навык подтверждён/отклонён, аттестация запрошена/завершена, сертификат добавлен вручную HR) — в `audit_log`: `user_id`, `action`, `entity`, `old_value`, `new_value`, `timestamp`, `performed_by`
