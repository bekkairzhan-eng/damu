### Доступ и роли

- Роли хранятся в Keycloak. В демо — в localStorage (`auth:roles`)
- Обычный сотрудник без роли `hr`/`admin` видит экран "Нет доступа" при попытке зайти в `/hr` или `/admin`
- Пользователь с несколькими ролями видит переключатель панелей в TopNav (кнопка между иконкой apps и аватаром)
- Сотрудник с ролью `hr` в сайдбаре HR-панели видит ссылки: "Мой профиль" и "Панель администратора"
- Сотрудник с ролью `admin` в сайдбаре Admin-панели видит ссылки: "HR-панель" и "Мой профиль"

## Архитектурные решения (для ТЗ)

### Кластеры (треки)
Система поддерживает несколько кластеров с разными карьерными путями и требованиями:
- **К1** — Инфраструктура и Констракшн (госконтракты, особые требования)
- **К2-Север / К2-Юг** — основная ветка
- **К2-International** — международные проекты, более сложные навыки

Каждый сотрудник принадлежит к одному кластеру.
Карьерный граф (какая должность ведёт к какой) настраивается Админом отдельно для каждого кластера.
Каталог навыков один для всех кластеров. Отличается только требуемый уровень навыка на конкретную должность в конкретном кластере.
Модель: `должность × кластер × навык → требуемый_уровень`
Пример: "Работа с госдокументацией" — К1 требует Продвинутый, К2 — Базовый или не требуется.
Карьерная карта рендерится динамически из конфигурации — `CAREER_GRAPH` в коде уйдёт в базу.

### Мультиязычность

На старте — только русский язык. Но бэкендер должен заложить архитектуру с учётом будущей локализации:
- Текстовые поля (названия навыков, должностей, описания) хранить в формате, поддерживающем переводы (например, отдельная таблица переводов или JSONB с ключами по языку)
- Планируемые языки интерфейса: RU, KZ, EN, UZ, UA, AZ

### Стек
- Keycloak — авторизация
- S3 — файлы (сертификаты, аватары)
- Kafka — события между сервисами
- Redis — кэш
- UnityBPM — оркестрация задач (запрос аттестации → задача HR)

### Философия: Damu vs UnityBPM
- **Damu** — витрина для сотрудника и хранилище данных развития. Сотрудник видит свой прогресс, планы, навыки, треки
- **UnityBPM** — вся операционка: задачи, апрувы, эскалации, напоминания, блокировки почты
- Сотрудникам и утверждающим **не нужно заходить в Damu для апрувов** — всё через BPM
- HR заходит в Damu только чтобы внести данные по итогам аттестации (что недотягивает, какой план)

### Жизненный цикл карьерного плана

- У сотрудника всегда **один активный карьерный трек** (одна целевая должность)
- **По умолчанию** план создаётся автоматически при первом входе: цель = следующая ступень по основной ветке (Foreman D → C → B → A)
- Сотрудник может сменить цель сам через карьерную карту → план пересчитывается под новую должность
- Предыдущий план сохраняется в истории (статус `completed` или `replaced`)

Статусы плана: `active` / `completed` / `replaced`

### Результат аттестации (что вносит HR)

После аттестации HR не создаёт отдельный план. HR вносит **текстовый комментарий по каждому навыку** который сотрудник не дотянул:
- `skill_id` + текст "что нужно подтянуть"
- Эти комментарии отображаются в карьерном плане рядом с соответствующим навыком
- Статус плана после аттестации **не меняется** — остаётся `active`
- Количество попыток аттестации не ограничено

### Флоу аттестации через UnityBPM
1. Сотрудник нажимает "Запросить аттестацию" в Damu → событие уходит в BPM
2. BPM по матрице определяет состав аттестационной комиссии, HR назначает участников
3. Задачи участникам уходят через BPM (почта, уведомления, эскалации)
4. Если не реагируют — BPM блокирует/эскалирует автоматически
5. По итогам HR вносит результат в Damu: что не дотянул, формирует план развития
6. Damu обновляет статус сотрудника и карьерный план

### Интеграции
- **HRMS** — источник правды для:
  - Профиль сотрудника (ФИО, фото, должность, стаж, подразделение)
  - Орг. структура (Компания → Дивизион → Организация → Проект)
  - Корпоративная активность (индекс согласования, участие в мероприятиях)
  - Данные для рейтинга профиля
  - Зарплатные вилки по должностям (берутся из HRMS, отображаются всем сотрудникам на странице Должности)
  - Проекты сотрудника: из HRMS подтягивается 1 (основной/текущий) проект

### Проекты сотрудника (ExperienceProfile)

Сотрудник одновременно может быть задействован на 2–3 проектах.
- **Из HRMS:** подтягивается 1 проект (основной). Только чтение, не редактируется.
- **Ручной ввод:** сотрудник добавляет остальные проекты сам (CRUD). Поля: название, даты, описание.
- Проекты — это портфолио/история. На рейтинг и навыки напрямую не влияют (пока).
- **UnityBPM** — вся операционная логика: матрица утверждающих, задачи, эскалации, состав комиссии
- **LMS** — собственный продукт. Когда сотрудник завершает курс, событие приходит в Damu и обновляет прогресс обучения в карьерном плане
- **SuperApp** — отдельный продукт, ещё не готов. Будет источником одного из компонентов рейтинга (Рейтинг SuperApp). Интеграция — позже, пока заглушка.

### Рейтинг профиля

Итоговый балл (например, 4.3 / 5.0) считается как взвешенная сумма 5 компонентов.

| Компонент | Источник данных | Примечание |
|-----------|----------------|------------|
| Подтверждённые навыки | Damu (внутренние данные) | Доля подтверждённых навыков от требуемых |
| Активность обучения | LMS (Kafka-событие) + ручной ввод сотрудника | Курсы, часы, сертификаты |
| Рейтинг SuperApp | SuperApp (интеграция позже, сейчас заглушка) | — |
| Заполненность профиля | Damu | Процент заполненных полей профиля |
| Корпоративная активность | HRMS | Индекс согласования, участие в мероприятиях |

**Веса компонентов** — хранятся в базе (не хардкод), настраиваются Админом. Текущие значения: 30/25/25/10/10 — временные, будут пересматриваться.

**Пересчёт:** 1 раз в сутки (ночной батч). Не в реальном времени.

**Скоуп ранжирования:** внутри одной Компании холдинга + одного кластера.
- Пример: Foreman B в К1 ранжируются только с другими Foreman B в К1 той же компании.
- K1 с K2-Север не смешиваются.

### Уведомления (через Kafka)
Два триггера для уведомлений в Damu:

1. **Апрув/отклонение навыка** — результат приходит из UnityBPM webhook → сотрудник получает уведомление в Damu
2. **Админ добавил навык в набор должности** — событие публикуется в Kafka → все сотрудники с этой должностью получают уведомление, счётчик навыков в плане обновляется (было 18 → стало 19), в плане появляется баннер "Содержимое вашего плана обновлено" (уже есть в макете)

### Раздел "Аттестация" (отдельный экран)

Пункт "Аттестация" в навигации — отдельная страница, пока заблокирована (скоро).

**Для сотрудника:** полная история всех аттестаций. На каждую запись:
- Дата подачи и дата завершения
- Общий комментарий HR
- По каждому навыку: прошёл / не прошёл + текстовый комментарий HR
- Сотрудник может подавать сколько угодно раз — все попытки хранятся

**Для HR:** вводит результаты аттестации:
- По каждому навыку: статус (прошёл / не прошёл) + текстовый комментарий "что нужно подтянуть"
- Общий комментарий по сотруднику

**Связь с планом развития:**
- "План развития на основе оценки" в разделе "Моё развитие" показывает **только последнюю** запись HR
- Показывает навыки которые не прошёл + комментарии HR по ним
- При каждой новой аттестации план обновляется автоматически

Таблица в БД: `assessments` — одна запись на попытку аттестации, связана с `assessment_skills` (навыки + результат + комментарий).

**"Мой наставник"** — заблокирован, бизнес не готов. В модели данных не закладывать.

### Роли в системе
| Роль | Что может |
|------|-----------|
| `employee` | Видит свой профиль, планы, навыки, карьерный трек. Запрашивает аттестацию |
| `hr` | Вносит результаты аттестации, формирует план развития по итогам, видит профили сотрудников |
| `admin` | Настраивает каталог навыков, должности, требования по кластерам, карьерный граф, зарплатные вилки |

Утверждение навыков (skill_approver) — не отдельная роль в Damu. Апрув происходит через UnityBPM, результат прилетает в Damu через webhook.

### Логика подтверждения навыков по уровням

| Уровень | Действие |
|---------|----------|
| 1 — Базовый | Сотрудник выставляет сам, апрув не нужен |
| 2 — Средний | Сотрудник выставляет сам, апрув не нужен |
| 3 — Продвинутый | Уходит на апрув специалисту через UnityBPM |
| 4 — Эксперт | Уходит на апрув специалисту через UnityBPM |

Файлы-доказательства не требуются. Апрувер принимает решение на основе своего профессионального суждения.

### Матрица утверждения навыков
- Один утверждающий на один навык
- Утверждающий определяется по двум измерениям:
  1. **Орг. структура**: Компания / Дивизион / Организация / Проект
  2. **Навык**: для каждого навыка настраивается кто его апрувит (РП, тех специалист, другой)
- Матрица в UnityBPM будет дополнена измерением по навыку — гибкая настройка
- Пример: "Управление субподрядчиками" апрувит РП, "BIM-технологии" апрувит тех специалист
- Логика хранится в UnityBPM, Damu передаёт `skill_id + employee_id` → BPM сам определяет утверждающего
- **История подтверждений хранится в Damu:** кто подтвердил, когда, решение (подтверждено / отклонено / комментарий)
- **Переназначение при увольнении** — через BPM, не через Damu. BPM хранит все активные задачи, массово переназначает их новому человеку. Damu просто получит результат через тот же webhook от нового утверждающего — никаких изменений на стороне Damu не требуется

## Модель данных (сущности)

> Черновик, выведен из логики выше. Требует ревью перед передачей бэкендеру — поля помеченные `?` под вопросом.

Все текстовые поля (`name`, `description`, `comment`) — с учётом мультиязычности (см. раздел выше): отдельная таблица переводов или JSONB `{ru, kz, en, ...}`.

| Сущность | Ключевые поля | Источник | Связи |
|----------|---------------|----------|-------|
| `employee` | id, ФИО, фото, position_id, cluster_id, org_unit_id, стаж, company_id | HRMS (read-only) | → position, cluster, org_unit |
| `position` | id, name, grade, description, salary_min?, salary_max? | 1С / HRMS (вилки) | — |
| `cluster` | id, name (K1 / K2-Север / K2-Юг / K2-International) | Damu | — |
| `career_graph_edge` | cluster_id, from_position_id, to_position_id | Damu (Admin) | → cluster, position ×2 |
| `skill` | id, name, category, approver_rule? | Damu (Admin) | — |
| `requirement` | position_id, cluster_id, skill_id, required_level (0–4) | Damu (Admin) | → position, cluster, skill |
| `employee_skill` | employee_id, skill_id, current_level, status (self / pending / confirmed / rejected), confirmed_by, confirmed_at | Damu | → employee, skill |
| `career_plan` | id, employee_id, target_position_id, status (active / completed / replaced), created_at | Damu | → employee, position |
| `plan_learning_item` | id, plan_id, type (курс / сертификат / рекоменд.), title, status (done / in-progress / not-started), source (LMS / manual) | LMS + ручной ввод | → career_plan |
| `assessment` | id, employee_id, target_position_id, cluster_id, submitted_at, completed_at, hr_comment, status (pending / completed) | Damu | → employee, position |
| `assessment_skill` | assessment_id, skill_id, result (passed / failed), comment | Damu | → assessment, skill |
| `skill_approval_history` | id, employee_id, skill_id, approver, decision, comment, decided_at | Damu (webhook из BPM) | → employee, skill |
| `rating_weight` | component (enum, 5 шт.), weight (%) | Damu (Admin) | — |
| `profile_rating` | employee_id, score, component_scores (json), computed_at | Damu (ночной батч) | → employee |
| `project` | id, employee_id, source (HRMS / manual), title, date_from, date_to, description | HRMS (1 осн.) + ручной | → employee |
| `notification` | id, employee_id, type (skill_approved / plan_updated), payload, is_read, created_at | Damu (Kafka) | → employee |
| `user_role` | employee_id, role (employee / hr / admin) | Keycloak | → employee |

Кластер сотрудника, орг. структура, зарплатные вилки — из HRMS, в Damu не редактируются.

## Требования ИБ

- **Роли проверяются на сервере.** `RoleGuard` на фронте — только UX (скрыть меню), не защита. Каждый эндпоинт валидирует роль из Keycloak-токена независимо от клиента.
- Эндпоинты `/api/admin/*` — только `admin`, `/api/hr/*` — только `hr`. Сотрудник не может дёрнуть чужой профиль (`/api/employees/:id` — проверка прав).
- Токен Keycloak — в httpOnly cookie либо в памяти, **не в localStorage** (XSS).
- HR видит профили сотрудников только в рамках своего скоупа (компания / кластер?) — уточнить.
- Аудит-лог действий admin/hr (кто, что, когда изменил): навыки, требования, роли, результаты аттестации.
- Файлы (S3) — доступ по подписанным ссылкам с TTL, не публичные бакеты.
- Персональные данные (ФИО, фото) — в логи не писать.

## Нефункциональные требования (НФР)

- **Масштаб:** ~4000 сотрудников. Админ-таблицы (пользователи, навыки) — пагинация + поиск на сервере, не отдавать всё разом.
- **Рейтинг:** ночной батч, не realtime (см. раздел «Рейтинг профиля»). Ранжирование — в разрезе компания × кластер × должность.
- **Кэш:** Redis для справочников (навыки, должности, граф, требования) — меняются редко, читаются часто. Инвалидация при правке админом.
- **Пустые / ошибочные состояния:** каждый экран с данными из API должен иметь состояния «загрузка», «пусто», «ошибка загрузки». Сейчас в прототипе их нет.
- **Браузеры:** актуальные Chrome / Edge (корп. среда). Мобильная адаптивность уже заложена (`useBreakpoint`).
- **Доступность:** ARIA / семантика / фокус — долг прототипа, заложить в бэклог.
